> 本文档是用户在 Phase 3 后做的产品方向重大升级。**这是所有下游文档的 ground truth**。后续任何文档若与本文档冲突,以本文为准。

> 旧定位:多 Agent 视角解读工具

---

## 1. 产品本质再定义

**议见(YiJian)是一个面向企业跨部门决策的 AI 共识形成系统。** 它把一份业务提案拆成"角色、事实、分歧、权重、证据链",最后产出**可复现、可解释、有方法论支撑**的决策结论。

### 它要解决的真问题(企业跨部门决策的 4 个老毛病)

1. **各说各话**:不同部门视角碎片化,缺乏共同对齐
2. **分歧难收敛**:讨论容易跑偏 / 被强势角色带偏 / 反复扯皮
3. **依据不透明**:结论拍脑袋,没人能复盘"为什么这么定"
4. **结果不稳定**:同一个问题不同会议不同结论,无可信度

### 它不是什么(明确避免误区)

- ❌ **不是普通多 Agent 辩论室** — 纯辩论容易随机/发散/相互迁就/被 CEO 带偏
- ❌ **不是角色扮演聊天工具** — 不追求"对话像真人"
- ❌ **不是会议总结工具** — 上游不抢 Zoom/Otter 的活
- ❌ **不是通用 AI 队友** — 不抢 Asana/Dust 的通用赛道

### 它真正的产品价值

**结构化提案 → 基于证据发言 → 识别分歧 → 动态加权 → 形成可解释共识。**

## 2. 核心机制:三层升级

### 升级 1:多 Agent 发表可溯源观点(基础层)

每个角色 Agent 必须:
- 基于**自己的上下文 + 证据链**发言(不能凭空说)
- 输出**可溯源**(提案原句 + 角色规则 + 引用的数据源)
- 标注**置信度**

### 升级 2:多观点 → 共识的工程方法论(核心层)

**四层共识框架**(这是产品的方法论骨架,所有 UI 流程围绕这 4 层组织):

#### L1 战略/目标共识

- 先明确**公司级目标**(增长 / 营收 / 品牌 / 现金流 / 年度重点)
- 所有角色观点必须先服从这个**大目标**
- 借鉴方法论:**Hoshin Kanri**(方针管理)、**OKR**、**Balanced Scorecard**
- 产品落地:每次推理前需要"目标对齐"步骤,让用户/系统先声明这次提案服从哪个公司级目标

#### L2 事实/证据共识

- 系统需要**数据/证据引擎**,接入内外部数据:
  - **内部**:飞书文档 / ERP / 仓储 / 历史项目文档 / 历史决议
  - **外部**:市场趋势 / 竞品 / 行业数据 / 公开知识
- 每个 Agent 的观点必须能追溯到**证据链**
- 借鉴方法论:**Issue Tree / MECE**(逻辑分解)、**A3 报告**(丰田式问题分析)
- 产品落地:**P11 证据库**页面 + **证据链卡片**组件

#### L3 利益/角色共识

- 不同角色有不同关注点和**天然冲突**:
  - 财务:资金安全 + 效率
  - 运营:增长 + 节奏
  - 供应链:供给能力 + 合规
  - 市场:声量 + 转化
  - 品牌:认知一致性
  - 商品:品类结构
  - **区域管理(新)**:跨地域差异 + 本地化
- 系统**结构化**这些角色立场,不是让它们自由发言
- 借鉴方法论:**Stakeholder Mapping**、**RACI / RAPID / DACI**、**Harvard Principled Negotiation**(原则谈判)
- 产品落地:**P05 角色 Persona 工坊**升级到含"利益边界"和"天然冲突点"字段

#### L4 权重/权责共识

- 不同场景下角色权重**动态变化**:
  - 营销决策 → 市场 / 品牌权重高
  - 预算决策 → 财务权重高
  - 选品决策 → 供应链 / 运营 / 市场权重均高
  - 经营决策 → 综合
- 借鉴方法论:**Vroom-Yetton-Jago**(决策风格)、**DACI**(责任矩阵)
- 产品落地:**P02 提案输入**自动识别决策类型 → **动态权重计算器**

### 升级 3:学术/算法层防"LLM 跟风"(可信度层)

**问题根源**:LLM 智能体在辩论里会**跟风、最后一票随机性、看起来一致 ≠ 真的对**。

**产品级解法**:

1. **独立先发表(Blind First-Vote)**
   - 每个角色 Agent 在不看其他角色输出的情况下,先发表初始观点
   - 之后再做"二轮调整"
   - **算法依据**:防止 anchoring bias

2. **TWS 轨迹加权评分(自研,启发自 MAD 文献)**
   - 不依赖"最后一轮投票",而是基于**整个辩论轨迹**评分
   - **启发来源**:Du et al. 2023 multi-agent debate / MAD-Bench / Asch 一致性实验
   - 完整算法和衰减系数(R0=0.6, R1=0.4)见 [consensus-algorithm.md § 对策 2](../03-tech-direction/consensus-algorithm.md)
   - 产品落地:共识算法层使用轨迹评分而非简单 majority vote

3. **Premortem(集体预想失败)**
   - 在出结论前,让所有角色集体预想"如果这个方案 6 个月后失败了,最可能的原因是什么"
   - 反向校验,逼出隐藏风险
   - 借鉴方法论:**Premortem**(Gary Klein)
   - 产品落地:决策报告"风险"部分必跑此步

4. **稳定复现性测试(Reproducibility Check)**
   - 同一问题运行 N 次(温度变化 / 不同随机种子)
   - 看**主要结论排序**和**关键分歧**是否一致
   - 这是**产品自检 KPI**:稳定复现率 ≥ **67%**(3 次中至少 2 次主要结论一致)才算"可信"
   - 产品落地:**P07 Safety Center 新增面板"稳定性测试"** + Demo 时主动展示

5. **After Action Review(事后复盘)**
   - 决议执行后,记录实际结果 vs 预测
   - 反哺 Persona 备注 + 调整未来权重
   - 借鉴方法论:**After Action Review**(美军方法)、**PDCA / PDSA**(质量管理)
   - 产品落地:**P09 决议录入**升级到 AAR 模板

## 3. 量化标度升级(3 档 → 4 档)

**旧**:支持 / 中立 / 反对(3 档)
**新**:**支持 / 谨慎支持 / 反对 / 信息不足**(4 档)

为什么加"谨慎支持":
- 中立 ≠ 谨慎支持 — 中立是真的没意见,谨慎支持是"原则上同意但有前置条件"
- 谨慎支持本身就是分歧识别的关键信号

为什么加"信息不足"(取代"无数据"):
- "无数据"是被动的(系统没要到)
- "信息不足"是主动的(角色明说"我需要 X 数据才能判断")→ 直接转化为待回答清单

**4 档颜色映射**(同步更新 ui.md):

| 档位 | 颜色 token | Lucide icon |
| --- | --- | --- |
| 支持 | `--success`(绿) | `CheckCircle2` |
| 谨慎支持 | `--conditional`(浅绿/青) | `CheckCircle` |
| 反对 | `--destructive`(红) | `XCircle` |
| 信息不足 | `--warning`(琥珀/黄) | `HelpCircle` |

## 4. 角色升级(6 → 7 角色)

| 角色 | 关注 | 天然冲突点 |
| --- | --- | --- |
| 运营 | 活动节奏 / 库存周转 / 流量预算 | vs 财务(花钱) / vs 供应链(节奏) |
| 商品 | 品类结构 / 新品占比 / 季节适配 | vs 市场(主推选什么) |
| 市场 | 品牌声量 / 定位匹配 / 价位带 | vs 财务(投流预算) |
| 财务 | ROI / 毛利 / 回款周期 | vs 所有(花钱) |
| 品牌 | 调性一致性 / 跨市场认知 | vs 运营(短期增长 vs 长期品牌) |
| 供应链 | 备货周期 / 产能 / 合规 | vs 运营(节奏) / vs 商品(SKU 数量) |
| **区域管理(新)** | 跨地域差异 / 本地化 / 区域 KPI | vs 总部(标准化 vs 本地化) |

用户也可以**作为 stakeholder 加入** — UI 给"我想自己发表观点"按钮(P0 P1 边缘,可选)。

## 5. 用户流程(7 步)

```
1. 用户输入业务提案
        ↓
2. 系统自动结构化提案 + 识别决策类型(选品 / 营销 / 经营 / 预算)
        ↓
3. 系统从角色库推荐参与角色 + 用户增删 + 用户可加入自己
        ↓
4. 各角色 Agent 基于上下文 + 证据发表观点(blind first-vote)
        ↓
5. 系统高亮分歧 + 4 档量化(支持/谨慎支持/反对/信息不足)
        ↓
6. 基于目标/事实/利益/权重做共识收敛(TWS 轨迹加权评分(自研) + Premortem)
        ↓
7. 输出决策报告:结论 / 评分 / 关键分歧 / 证据链 / 风险 / 建议行动 / 纪要
```

---

## 6. 决策报告 7 部分(P12 新页 / 或 P06 升级)

1. **结论**(通过 / 暂缓 / 驳回 / 需补数据)
2. **评分**(加权后总分 + 4 档支持率分布)
3. **关键分歧**(top 3,带角色立场对照)
4. **证据链**(每条结论引用的内外部数据 + 文档 + 历史案例)
5. **风险**(Premortem 产出 + 风险等级)
6. **建议行动**(RACI / DACI 责任分配 + 时间节点)
7. **纪要 / 共识文本**(可直接发飞书 / 微信的会议纪要 + 链接)

## 7. Demo 场景聚焦

**确认场景**:**电商 / 珠宝选品决策**(七夕 / 520 / 季度主推)

**为什么选这个**:
- 高频(电商每月都有大促)
- 多角色冲突天然(市场/品牌/财务/运营/供应链都关心)
- 决策结果短期可验证

**不做**(明确划界):
- ❌ "总部-区域-门店全链路决策" — 链路太长 / 频率不够 / 复杂度失控
- ❌ 只在选品场景里**轻量体现"区域管理"角色**(证明系统考虑了组织分层即可,不深入做)

## 9. 对评审打分的强化(增量)

新方法论对评审打分的**增量影响**:

| 评审维度 | 新增证据 |
| --- | --- |
| 业务落地 25% | **四层共识框架 + 5 个 P0 真落地方法论**(OKR / Stakeholder Mapping / RACI / Premortem / AAR,每个都有 schema 字段 + UI 组件 + prompt 注入)= 不是"我们做了个 AI 工具",而是"我们工程化了经典企业决策方法论"(详见 [methodology.md](../03-tech-direction/methodology.md);8 个 V2 候选作为路线图) |
| 技术实现 20% | **TWS 轨迹加权评分算法(自研,启发自 MAD) + Blind First-Vote + 稳定复现测试** = 学术级严谨度 |
| 风险安全 20% | **稳定复现测试**新面板 + **Premortem 风险预想** = 风险机制再深一层 |
| 创新性 15% | **方法论 + AI 算法 + 工程化** 三重创新,非简单"AI 包装" |
| 演示 10% | **稳定性 Live 测试**可演示(同问题 3 次运行,结果对比) |
| 非技术可理解 10% | **四层共识** + **决策报告 7 部分** 都是业务语言,非技术 jargon |
