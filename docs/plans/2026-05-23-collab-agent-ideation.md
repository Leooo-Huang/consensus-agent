**议见把一份业务提案拆成"角色、事实、分歧、权重、证据链",输出可复现、可解释、有方法论支撑的决策结论。**

## 本质任务分析 (v1 历史,留作审计)

### 本质任务

**`一份单一视角的选品提案文本 → [角色化解读 + 分歧识别 + 结构化归集] → 一份多视角对齐的可执行讨论框架`**

更精炼一句:**把"一个人的判断"变成"一群人的共识起点"。**

### 核心价值链(系统视角)

1. **提案摄入** — 接收文本/语音/图片(选品理由、市场观察、业务假设)
2. **结构解构** — 把自由文本拆成可验证的"主张 / 依据 / 假设 / 数据缺失"四元结构
3. **角色建模** — 装载 N 个角色 persona(目标、KPI、口头禅、风险偏好)
4. **多视角推理** — 每个角色对每个主张/依据/假设独立打分 + 提出关注点
5. **分歧识别** — 跨角色聚类相同关注点 → 浮出共识区 / 分歧区 / 盲区
6. **讨论框架生成** — 输出一份带"待回答问题"的结构化议程
7. **会议沉淀回写** — 决议/补充数据/角色立场变化沉淀到知识库,下次复用

### 用户旅程

1. 提案人粘贴/上传选品文本
2. 选择/确认参与角色(默认 6 个,可加减)
3. 等待 Agent 解构 + 多视角分析(秒级)
4. 看到"主张矩阵 × 角色热力图"
5. 一键查看每个角色的"赞成 / 担忧 / 必须回答的问题"
6. 编辑/补充自己的回应
7. 把"待回答清单"分配给真人(可选)
8. 真人答复回填,Agent 重算共识度
9. 会议时直接打开"结构化讨论框架"投屏
10. 决议确认后归档,下次类似提案时角色 persona 自动学习更新

**交叉对照发现:**

- **用户旅程有 / 价值链缺**:第 2 步角色配置、第 6 步用户编辑回应、第 9 步会议投屏视图 → 都补到功能里
- **价值链有 / 用户不可见**:第 7 步会议沉淀回写(后台学习闭环)

### 失败模式

| # | 失败模式 | 后果 |
| --- | --- | --- |
| F1 | 角色推理变"刻板印象"(财务永远只关心 ROI) | Agent 失去价值,业务方失去信任 |
| F2 | Agent 编造数据冒充"市场角色担忧" | 误导决策 + 业务事故 |
| F3 | 提案涉及商业机密被 LLM 拿去训练 | 商业泄密 + 合规风险 |
| F4 | 推理失败/超时,提案人卡住 | Demo 翻车 + 真实场景失去用户 |
| F5 | Agent 输出过于主观,导致团队迷信 AI 结论 | 决策质量下降,责任错位 |
| F6 | 多角色输出长篇大论,业务人员读不完 | 工具被弃用 |

### 成功标准

- 同一份提案的决策周期从 N 天降到 1 次会议
- 会议时长缩短 ≥ 30%(因为分歧已经被前置识别)
- 业务方对 Agent 输出的"是否帮我节省了时间"打分 ≥ 4/5

### 直觉方案差距

- **直觉方案**:"把提案给 LLM,让它扮演 N 个角色各说一段。"
- **漏掉的环节**:
  - 结构解构 — 没拆颗粒,角色之间没法对齐
  - 分歧识别 — 只有"各说各的",没有"哪些点真的有分歧"
  - 可溯源 — 没有 citation 就是黑盒
  - 会议沉淀 — 永远是初见,无学习闭环
  - 风险/合规 — 直接把企业数据丢给云端 LLM,无脱敏 / 无 ZDR

## 产品定位

**CollabSense — 给企业电商运营的"提案 → 共识"AI 协作中枢。** 让一份单视角的选品提案在 30 秒内变成多角色对齐的可执行讨论框架,显著缩短跨部门决策周期。

## 调研发现

### 用户洞察

- 决策周期长的根本原因不是"会议多",而是"会议时大家对论点的颗粒度都不一致"
- 业务人员对 AI 工具的信任度极度依赖"我能看到它说的依据在哪"
- 跨文化协作场景下,角色 persona 还涉及"工作风格"(如中日韩在风险偏好上差异显著)

### 前沿研究

| 论文/工具 | 可转化为功能 |
| --- | --- |
| **LangGraph 0.4 HITL checkpoints** | 任意一步可人工接管,直接对应风险机制 20% |
| **TWS(自研轨迹加权评分) 共识轨迹评分** | 分歧度算法基础 |
| **Hermes Agent 5 种投票策略** | 共识聚类参考 |
| **CrewAI role/goal/backstory** | Persona 配置直接模式参考 |
| **Vercel AI SDK v6 + AI Gateway** | 多 Provider 路由 + 降级链开箱即用 |

#### P1 LLM Provider 路由 + 降级链
- **用户故事**:作为系统,我可以在 Opus 4.7 → Sonnet 4.6 → Haiku 4.5 → 离线规则之间自动切换
- **核心交互**:每次推理超时 / 报错 / 限流 → 自动降级,UI 角标显示当前 Provider
- **技术关键词**:Vercel AI Gateway、AI SDK v6 provider strings、熔断器(opossum 风格)
- **隐性需求覆盖**:符合用户全局规则"禁止静默 fallback" — 降级必须可见

#### P2 结构化输出引擎
- **用户故事**:作为系统,我可以强制 LLM 输出符合 Zod schema 的 JSON,以便下游严格使用
- **核心交互**:无 UI,后端能力
- **技术关键词**:AI SDK `generateObject`、Zod、JSON Schema

#### P3 数据脱敏管线
- **用户故事**:作为系统,我可以在提案进入 LLM 前剥离敏感字段(供应商名/客户名/成本数字),输出后还原
- **核心交互**:在 W4 Safety Center 显示"脱敏前 vs 脱敏后"对比
- **技术关键词**:正则 + 占位符映射 + 反向还原表
- **隐性需求**:可溯源 — 脱敏映射要可审计

#### P4 Persona / 决议存储
- **用户故事**:作为系统,我可以持久化角色 persona 和会议决议,以便复用和学习
- **核心交互**:Persona 工坊 + 决议列表
- **技术关键词**:SQLite(Demo)/ Postgres(生产)+ Drizzle ORM

#### P5 可视化引擎
- **用户故事**:作为系统,我可以渲染"主张 × 角色"矩阵 + 热力图 + 关注点 hover 高亮
- **核心交互**:W3 分歧热力图视图的底层
- **技术关键词**:Recharts / Visx、Tailwind、framer-motion

#### W1 提案推理流水线(含 F1+F3+F6+F5+F8)
- **用户故事**:作为运营经理,我贴一段选品提案,30 秒内拿到 6 角色解读 + 分歧 + 待回答清单 + 三句话摘要,每条都可溯源
- **核心交互**:左侧粘贴文本 → 中间实时流式输出"原子论点拆解 → 角色推理 → 分歧聚合 → 议程生成" → 右侧"一句话核心分歧"卡片
- **关键交付物**(同一次推理):
  1. 主张/依据/假设/数据缺失 四元结构
  2. 6 角色 × N 论点 的态度矩阵
  3. 共识 / 分歧 / 盲区 三色分组
  4. 待回答问题清单(Owner 待指派)
  5. 三句话摘要 + 一句话核心分歧
  6. 每条结论的 citation(提案原句 + 角色规则)
- **技术关键词**:LangGraph 并发 node(Send API)、AI SDK v6 streamObject、Zod schema 强约束、可溯源 metadata 强制字段

#### W2 角色 Persona 工坊
- **用户故事**:作为业务负责人,我可以查看/编辑 6 个角色 persona(运营/商品/市场/财务/品牌/供应链),适配本公司分工
- **核心交互**:卡片网格 → 点击展开 → 表单编辑(目标、KPI、风险偏好、决策口头禅、近期决议链接)
- **6 个内置 Persona 草稿**:
  - **运营**:在意活动节奏、库存周转、流量预算
  - **商品**:在意品类结构、新品占比、季节适配
  - **市场**:在意品牌调性、声量分布、价位带
  - **财务**:在意 ROI、毛利、回款周期
  - **品牌**:在意定位一致性、消费者认知、跨市场差异
  - **供应链**:在意备货周期、产能、合规
- **技术关键词**:React Hook Form + Zod、shadcn/ui Card + Drawer

#### W3 分歧热力图视图 ⭐
- **用户故事**:作为决策者,我打开一个矩阵就能看出"哪些点共识、哪些点雷区",不需要读长文
- **核心交互**:
  - X 轴 = 提案中的"原子论点"
  - Y 轴 = 6 个角色
  - 单元格颜色 = 支持(绿)/ 中立(灰)/ 反对(红)/ 无数据(白)
  - Hover 单元格 → 弹出该角色对该论点的关注点 + citation
  - 顶部"分歧热度排序"按钮 → 按"红绿混杂度"降序排列论点
- **技术关键词**:Recharts HeatMap / 自定义 SVG + Tailwind、framer-motion 过渡
- **创新点**:**市面无类似产品** — 这是产品的视觉标签

#### W4 风险护栏中心 ⭐
- **用户故事**:作为合规/管理者,我有一个统一面板看到所有风险机制实时状态
- **核心交互**(一个页签栏):
  1. **降级链状态** — 当前 Provider + 最近 10 次降级记录
  2. **数据脱敏对比** — 本次推理"脱敏前 vs 脱敏后"diff
  3. **置信度阈值** — 本次输出的逐条置信度,< 0.6 标红"低置信"
  4. **回滚历史** — 每次推理版本号,一键回到上一版
  5. **HITL 入口** — 当前是否处于"等待人工审核"状态,审核者可在此放行/驳回
  6. **审计日志** — 时间戳 / 操作人 / 输入 hash / 输出 hash
- **技术关键词**:LangGraph interrupt() + checkpoint、Tailwind Tabs、Zustand 状态管理
- **创新点**:**做成可视化中心**,而不是埋在配置文件里 — **评审一眼能看到 20% 风险分**

#### W5 决议录入(F9 mini)
- **用户故事**:作为提案人,会议结束后我可以录入决议(通过/暂缓/驳回 + 关键改动),Agent 把结论附到对应 Persona 备注
- **核心交互**:从讨论框架页 → "录入决议"按钮 → 表单 → 保存到 Persona 历史
- **技术关键词**:简单表单 + Persona 关联表
- **范围限制(MVP 内)**:**只做手动录入 + 写入 Persona 备注**,不做 RAG 自动校准(留 V2)

#### W6 Demo 模拟器
- **用户故事**:作为路演者,我可以一键切换 4 个预置选品场景,Demo 全程不卡壳
- **4 个预置场景**:
  1. **小红书声量提案**(原赛题示例,主推 A 款项链)
  2. **节日大促提案**(七夕情侣对戒主推方案,有强烈节奏冲突)
  3. **高客单旗舰款提案**(百万级单品,财务 vs 品牌 vs 供应链高度分歧)
  4. **跨境市场新品提案**(同一款 SKU 推日本/东南亚两个市场,跨文化角色冲突)
- **核心交互**:首页右上"演示模式"开关 → 下拉选场景 → 一键加载提案文本 + 预设角色配置
- **技术关键词**:JSON 预置文件 + 路由参数

#### W7 评审视角对照页 ⭐
- **用户故事**:作为评审,我打开一页就能看到"6 项打分点 ↔ 产品功能"的映射,不需要自己琢磨
- **核心交互**:一个长页面,6 个 section 对应 6 项评审维度,每个 section 列:
  - 该维度的权重和定义
  - 本产品如何拿这个分(具体功能 + 截图入口)
  - 一键跳转到对应功能页演示
- **技术关键词**:Tailwind 长页 + 锚点导航
- **创新点**:**Hackathon 特化招式** — 不是产品功能,是"打分动线优化"

## MVP 总结

**为什么这个 MVP 组合有意义:**

- **平台层 P1-P5 是地基**:每一层都直接对应失败模式 + 评审风险机制 20%(P1 降级 / P3 脱敏 / P4 持久化让一切可审计 / P5 可视化让一切可被看见)
- **场景层 W1-W7 是 65% 评审分的执行**:
  - W1+W2+W3 = **业务落地 25%**(端到端真实流程)
  - W4 = **风险机制 20%**(可视化中心,不是文档里一句话)
  - W3 + W6 = **创新 15%**(分歧热力图 + 黑客松场景模拟器)
  - W6 + W7 = **演示 10% + 非技术可理解 10%**(动线 + 评审 cheatsheet)
- **W7 是"打分动线设计"**:把评审从"自己琢磨"变成"被产品引导",这是大多数选手不会做的反向工程

## 🎯 评审得分点 × 创意映射表(核心章节)

> 这是本方案的灵魂:**每个评审维度都不是事后凑数,而是产品从架构层就主动设计去拿分。**

### ① 业务落地潜力 — 25%(最大权重)

**评审关注**:解决真实痛点、可落地、有价值。

| 创意切入点 | 怎么拿分 |
| --- | --- |
| **垂直绑定企业 X 真实场景** | 不用通用 SaaS 话术,Demo 全程围绕"珠宝零售选品提案",连 4 个预置场景都贴合企业 X 业务(小红书声量 / 七夕情侣对戒 / 百万旗舰 / 跨境市场) |
| **角色 Persona 来自真实分工** | 6 个角色直接对应企业 X 多职能协同(运营/商品/市场/财务/品牌/供应链),不是抽象的"专家 1/2/3" |
| **可量化的成功指标** | 文档明确:决策周期从 N 天 → 1 次会议,会议时长缩短 ≥ 30% |
| **学习闭环(W5)** | 决议录入证明产品不是"一次性玩具",而是"越用越懂这家公司"的资产 |
| **跨文化扩展路径(V2 P6)** | 企业 X 横跨中日韩美东南亚,产品规划已经包含 Persona 风格适配 |

### ② 技术实现 — 20%

**评审关注**:代码质量、架构设计、系统可运行性。

| 创意切入点 | 怎么拿分 |
| --- | --- |
| **架构清晰分层**(P/W 双层) | 平台能力 P1-P5 与场景能力 W1-W7 严格解耦,符合 MECE |
| **生产级技术栈**(Next.js + AI SDK v6 + LangGraph 0.4) | 不堆砌实验性技术,选每个组件都有官方文档背书 |
| **AI Gateway 解耦 Provider** | 不绑死单一 LLM 厂商,2 行代码可换模型,符合企业级实践 |
| **强 Schema 约束**(Zod + generateObject) | 每个 LLM 输出都是结构化数据,不靠"prompt + regex 拼凑" |
| **可观测性内置**(W4 审计日志) | 不是事后加,是 Day 1 设计的一部分 |
| **类型安全 + 测试** | TypeScript 严格模式 + vitest 单元测试 + Playwright E2E(W6 4 个场景全跑通) |

### ③ 风险安全机制 — 20%(与"技术实现"并列第二大权重)

**评审关注**:异常监测、降级策略、人工接管、回滚机制。

| 创意切入点 | 怎么拿分 |
| --- | --- |
| **W4 风险护栏中心 = "评审一眼看到"** | 6 项风险机制做成一个可视化中心:降级链状态 / 脱敏对比 / 置信度 / 回滚历史 / HITL 入口 / 审计日志 — 不是配置文件里的一句话 |
| **降级链显式可见**(P1) | 当前 Provider 角标实时显示,降级到下一档时 UI 弹出 toast — 符合用户全局规则"禁止静默 fallback" |
| **数据脱敏 diff 视图**(P3) | "脱敏前 vs 脱敏后" 并排展示,reviewer 可以亲眼验证商业机密被剥离 |
| **置信度阈值显式标注** | 每条结论都有 confidence,< 0.6 自动标红"低置信,建议人工复核" |
| **回滚机制做到 UI**(W4) | 每次推理是一个版本,一键回到上一版 — 不是 git revert,是产品级回滚 |
| **HITL checkpoint**(LangGraph 0.4) | 任意一步可暂停等待人工放行,Demo 时主动展示这个能力 |
| **失败模式 → 功能映射**(F1-F6 → P/W) | 文档明确每个失败模式有哪个功能防护 — 评审看到这张表就知道我们想过 |

### ④ 创新性 — 15%

**评审关注**:方案切入点新颖,具备独特性。

| 创意切入点 | 怎么拿分 |
| --- | --- |
| **W3 分歧热力图 = 视觉锚点** | "主张 × 角色 矩阵"是市面没有的视觉表达,Demo 截图就是 SNS 传播图 |
| **结构化原子论点拆解**(F1) | 不是"让 LLM 各扮演角色说一段",而是先把提案拆成"主张/依据/假设/数据缺失",再让角色对齐颗粒度打分 — 思路新颖 |
| **可溯源到角色规则**(F5) | 每条结论都标"基于提案哪一句 + 哪条 Persona 规则",这种"角色规则 + 文本依据"的双重 citation 是 TWS(自研轨迹加权评分) 论文级别的工程化 |
| **W7 评审视角对照页** | **打分动线本身是创新** — 把评审从"自己琢磨我们做了啥"变成"被产品主动引导",大多数选手不会想到 |
| **学习闭环不是 RAG**(W5 → V2) | 决议直接回写到 Persona 备注,使 Agent 真正"内化"该公司的决策风格 |

### ⑤ 演示与表达 — 10%

**评审关注**:路演清晰、重点突出。

| 创意切入点 | 怎么拿分 |
| --- | --- |
| **W6 Demo 模拟器 = "不会翻车的演示"** | 4 个预置场景一键切换,网络/LLM 故障都有降级兜底 |
| **场景递进设计** | 4 个场景按"分歧强度"递增:小红书声量(轻冲突) → 节日大促(节奏冲突) → 百万旗舰(财务 vs 品牌强冲突) → 跨境新品(跨文化最强冲突) |
| **路演 5 分钟动线**(明文): | 1) 输入提案 30 秒 → 2) 看分歧热力图 30 秒 → 3) 展开角色关注点 60 秒 → 4) 主动演示降级 30 秒 → 5) 主动演示 HITL 接管 30 秒 → 6) 录入决议 30 秒 → 7) Safety Center 巡视 60 秒 → 8) 评审视角对照页收尾 30 秒 |
| **"一句话核心分歧"卡片**(F8 在 UI 顶部) | 路演时永远一眼能看到"这次的关键焦点是什么" |

### ⑥ 非技术可理解度 — 10%

**评审关注**:非技术人员易于理解。

| 创意切入点 | 怎么拿分 |
| --- | --- |
| **F8 三句话摘要 + 一句话核心分歧**(强制顶部呈现) | 任何复杂输出都先给"业务人话",再给细节 |
| **W3 热力图替代长文** | 颜色矩阵比"5 份评审意见"易读 100 倍 |
| **角色用"口头禅"表达立场** | 不是"该角色的关注维度是 ROI",而是 "财务说:你的客单价对得起这次活动投入吗?" — 评审听得懂 |
| **W7 评审视角对照页** | **就是为非技术人员写的导览** — 不需要懂 LangGraph 也能看明白产品在做什么 |
| **零 Jargon 命名** | "分歧热力图" / "风险护栏中心" / "讨论框架" 而不是 "Multi-agent debate consensus dashboard" |
